**临时联盟设计**只解决了其中 1 个（"长期合作伙伴议程冲突"）。剩下 5 个仍是 Mycelium 必须正面应对的。

## 失败原因 1：UI/UX 笨拙

### Plenti 当年具体做错了什么

| 痛点 | 详情 |
|------|------|
| **结账时报手机号** | 客户在 Macy's / AT&T 收银台必须报手机号或刷卡才能赚分；现金顾客几乎被排除 |
| **没有移动端 app（前两年）** | 直到 2017 年才推出 mobile app；之前用户必须记住物理卡号 |
| **App Store 搜索不到** | 搜 "Plenti" 出现各种无关 app（Plenti rapidash？plenti pump？）—— 用户找不到正确入口 |
| **不同合作伙伴流程不一致** | Macy's 自动累积、AT&T 月底批量、Hulu 立即——用户不知道什么时候该收到分 |
| **收银员不懂或不愿告知** | Macy's 员工被绩效驱动 push Star Rewards 而非 Plenti，常常"忘记"提醒客户 |
| **退分流程复杂** | 退货时如何调整积分没有统一规则；用户经常 "退了货积分还在" 或反向 |

### 为什么这是真问题（行为经济学）

1. **结账时间敏感**：消费者在收银台多花 30 秒就开始烦躁，Plenti 的多步操作直接拉低 NPS
2. **现金顾客（亚洲市场大头）**：Plenti 美国设计排除了 30% 现金消费场景；亚洲市场（日本/韩国/中国）现金 + 手机支付混合，纯线上方案行不通
3. **多平台切换疲劳**：用户在 19 个 loyalty 程序之间切换的成本随程序数线性增长

请你回答以下 6 个问题，每个 1-3 句话即可：

| # | 问题 |
|---|------|
| **Q1.1** | 用户在咖啡店结账时**怎么"用积分"或"赚积分"**？是否需要打开 app？是否需要扫码？是否需要授权？ |
| **Q1.2** | **现金顾客**怎么参与？还是 Mycelium 干脆放弃现金顾客？ |
| **Q1.3** | 商家收银员需要做什么？需要培训吗？POS 如何集成？ |
| **Q1.4** | 用户**手机里的钱包是 Sin90 / 商家自己 app / WhatsApp 小程序**？哪一个是默认入口？ |
| **Q1.5** | 退货 / 退款时积分如何处理？智能合约能否自动处理？ |
| **Q1.6** | 我们要做**自己的钱包 app**还是**让商家集成进自己的 app**？这个决策决定了产品形态 |

**关键决定**：UI/UX 不是"做得漂亮"的问题，是**Mycelium 是 app 还是 SDK**的根本架构选择。

## 失败原因 2：品牌认知度低

### Plenti 当年具体做错了什么

| 痛点 | 详情 |
|------|------|
| **品牌名无含义** | "Plenti" 既不是英文单词，也无产业背景；用户记不住 |
| **没有标志性合作伙伴** | Macy's / AT&T 都是巨头，但都不是"loyalty 标杆"——没有 American Airlines 这种"看到就想到 loyalty"的伙伴 |
| **营销预算分散** | Plenti 自己没投广告；合作伙伴各自营销自家程序，Plenti 在合作伙伴的营销中被边缘化 |
| **认知调研结果** | 2017 年调研：50% 美国人不知道 Plenti 是什么；自称会员的人 <50% 真正用过 |
| **搜索引擎无存在感** | 在 Google 搜 "loyalty rewards"、"multi-merchant rewards"、"earn points everywhere"，Plenti 都在第 3 页之后 |
| **媒体覆盖少** | 除了关闭时上了一波负面新闻，几乎没有产业媒体的深度报道 |

### 为什么这是真问题

1. **联盟天生品牌弱**：多商家共用一个品牌 = 没有强信任锚点
2. **平台型业务的 chicken-and-egg**：商家说"用户少不加入"，用户说"商家少不参与"
3. **营销 ROI 难分摊**：Plenti 的广告投出去，但不知道带来的客户去 Macy's 还是 AT&T，无法归因 → 没人愿意投

| **Q2.2** | 用户感知到的品牌是 Mycelium 自己，还是商家自己？类比：Visa 让商家挂"用 Visa"还是隐身？ |

## 失败原因 3：不对称兑换池

### Plenti 当年具体做错了什么

**赚分 vs 兑换的极度不对称**：

| 类别 | 商家 | 赚分难度 | 兑换难度 |
|------|------|---------|---------|
| 加油 | Exxon Mobil | ⭐ 极易（每天加油都能赚） | - |
| 杂货 | Direct Energy | ⭐ 极易 | - |
| 餐饮 | Chili's | ⭐⭐ 中 | ⭐⭐ 中（一顿饭） |
| 旅行 | Expedia | - | ⭐⭐⭐⭐⭐ 极难（需要数千分换 1 晚酒店）|
| 电信 | AT&T | - | ⭐⭐⭐⭐ 难（账单抵扣需要数百分换 $5）|
| 流媒体 | Hulu | - | ⭐⭐⭐⭐ 难 |

**结果**：用户加 1 年油 + 买 1 年杂货能攒到 3,000-5,000 分；最便宜的有意义兑换是 AT&T $20 抵扣（需 4,000 分）。**用户感觉积分"没价值"**。

### 为什么这是真问题（行为经济学）

1. **Frequent small wins 原则**：人类需要每周/每月可见的积分使用，不是 1 年 1 次大奖
2. **Earn:Burn ratio**：经过研究最有效的比例是 **3:1 到 5:1**（赚 3-5 次能花 1 次）；Plenti 实际是 50:1，导致流失
3. **价值感知陷阱**：当用户觉得"攒了这么久还不够花一次"，会自我证明"这玩意儿没用"，进而停止参与
4. **品类不对称**：日常品类（加油/杂货）赚分容易但兑换在低频品类（旅行/电信账单）— 用户行为路径割裂

### Mycelium 必须回答的问题

| # | 问题 |
|---|------|
| **Q3.1** | Mycelium 联盟形成时，**如何确保 earn:burn ratio 平衡**？谁来强制？ |
| **Q3.2** | 协议层是否有**最低兑换门槛**（如最低 50 积分可兑）？这个门槛由谁决定？ |
| **Q3.3** | AI 优化器**是否能预防**这种不对称？输入是什么、输出给谁、谁执行？ |
| **Q3.4** | 临时联盟（如"圣诞市集"3 个月）是否反而**有助于**对称——因为短期内频繁互动？ |
| **Q3.5** | 用户能否在 Sin90 钱包里看到**"我的积分还能兑什么"** 的实时清单？这是产品功能还是协议层？ |
| **Q3.6** | 如果某个商家发分太多（earn 过容易），其他商家利益受损，**Mycelium 是仲裁方还是中立协议**？ |

**关键决定**：earn:burn 平衡是**产品设计的核心 tension**。要么 Mycelium 主动管理（中心化），要么协议自动执行（去中心化）——这是哲学选择。

## 失败原因 4：UX 多步骤

### Plenti 当年具体做错了什么

| 痛点 | 详情 |
|------|------|
| **独立 app** | Plenti 是独立 app，与商家自有 app（Macy's app, AT&T app, Hulu app）不集成 |
| **多平台切换** | 用户在 Hulu 内不能直接看到 Plenti 余额；要切到 Plenti app |
| **结账时切换** | 商家收银台扫 Plenti 卡 → 切到商家自己流程 → 用户拿出商家卡——多步骤 |
| **邮件确认** | 每次赚分都发邮件 + Plenti app 通知 + 商家短信——信息过载 |
| **跨商家结算延迟** | 在 A 商家赚分，要 24-72 小时才能在 B 商家用——用户当场不明白 |
| **vs Amex MR** | Amex MR 在 Amex app 内一站式：查余额 / 转分 / 兑换 / 看历史，体验极简 |

### 为什么这是真问题

1. **App fatigue**：用户已经有太多 app（19 个 loyalty 程序）
2. **Cognitive load**：多 app 切换 = 体验断裂 = 用户流失
3. **商家不愿教育用户**：Macy's 不愿教用户用 Plenti（Plenti 蚕食 Macy's 自家 Star Rewards）
4. **跨平台数据同步**：实时同步是技术挑战，延迟即用户不信任

### Mycelium 必须回答的问题

| # | 问题 |
|---|------|
| **Q4.1** | 用户**有没有 Sin90 钱包就能用**还是必须装 Sin90？商家 app 内嵌还是独立？ |
| **Q4.2** | 如果商家 app 内嵌，**Mycelium 给商家的是 SDK / API / Web component**？哪种？ |
| **Q4.3** | **跨商家结算的延迟**接受多久？秒级（区块链 finality）/ 分钟级 / 小时级？ |
| **Q4.4** | 用户在结账时是 Sin90 出示 QR 还是商家扫用户钱包？哪个对**收银员零培训**最友好？ |
| **Q4.5** | **微信生态**怎么处理？小程序内嵌还是独立 app？中国早期不做用户但海外华人用户需要 |
| **Q4.6** | 类比：Mycelium 是更像 **Stripe**（embedded checkout，商家不感知）还是 **Apple Pay**（独立钱包，商家集成）？ |

**关键决定**：Mycelium 是**SDK 而非 app** 还是**app 而非 SDK**？这个决策一旦确定，后续全部产品形态都跟着走。

## 失败原因 5：商家 ROI 不清晰

### Plenti 当年具体做错了什么

| 痛点 | 详情 |
|------|------|
| **费率结构复杂** | 商家费用 = 月固定费 + 每笔积分发行费 + 每笔兑换费 + 平台年费——没人能算清 |
| **没有标准 ROI dashboard** | 商家自己查报表：发行了 X 分，兑换了 Y 分，破损率 Z，但无法回答"我从 Plenti 净赚多少" |
| **跨商家归因不清** | 用户在 A 商家赚分，B 商家兑换——B 商家"亏"，A 商家"赚"，谁补偿谁？ |
| **没有同行对比** | 商家不知道"我家的兑换率 vs 同行平均"——无法判断是否值得继续 |
| **退出成本高** | 商家想退出 Plenti，已发的积分还要兑现——退出 = 现金流出 |
| **Amex 内部沟通混乱** | Plenti 团队 vs Amex 自家产品团队互相争夺 = 商家收到混乱信号 |

### 为什么这是真问题

1. **中小商家不会精算**：老板不是 CFO，需要"一眼看明白"的数字
2. **决策依据不足 = 流失**：没有 ROI 数据 → 商家自然怀疑 → 找借口退出
3. **同行对比是激励**：大商家会与同行比，看到"我比平均差"才会努力
4. **退出成本陷阱**：进入容易退出难 = 商家不敢加入

### Mycelium 必须回答的问题

| # | 问题 |
|---|------|
| **Q5.1** | 商家 dashboard 显示的**核心指标**是什么？建议 3-5 个数字，要让面包店老板 5 秒看明白 |
| **Q5.2** | "**净 ROI**"如何计算？发行成本 vs 锁客带来的复购增量 vs 现金流贡献——谁追踪？ |
| **Q5.3** | **同行对比**怎么提供？需要聚合数据（你之前提到的 federated 数据）—— 隐私如何保护？ |
| **Q5.4** | 商家**退出联盟**时，未兑现的积分怎么处理？是 Mycelium 兜底（保证金机制）还是商家继续兑付？ |
| **Q5.5** | Performance-based fee 模式下，**Mycelium 怎么算"成交"**？发行积分算？兑换算？还是用户实际带来的复购销售？ |
| **Q5.6** | **AI 优化器**是否就是这个 ROI 工具的核心？建议商家"按你的客户结构，发 X% 的积分"——这是销售卖点也是合规边界 |

**关键决定**：商家 ROI 不是"附加功能"，是 Mycelium 留住商家的**唯一长期理由**。如果 ROI 不清，再好的协议设计都留不住商家。

## 综合：Plenti 教训给 Mycelium 的 6 个跨主题决定

回答完上面 5 个原因下的问题后，请你回答这 6 个**贯穿性决策**：

### Q.X1: Mycelium 是 app 还是 SDK？

- **App 模式**：用户装 Sin90 钱包，商家在自己 POS 集成支付/兑换接口
- **SDK 模式**：用户不装任何东西，每个商家自己 app 内嵌 Mycelium 模块（用户感知不到 Mycelium）
- **混合模式**：Sin90 是默认钱包，但商家可以白标用自己 app

### Q.X2: 中心化 vs 去中心化的边界在哪？

- 协议层（智能合约）= 去中心化
- 联盟匹配 / earn:burn 平衡 / 仲裁 = ?
- AI 优化器 = ?
- 商家 onboarding KYC = ?

### Q.X3: 积分价值的"锚定"

- Plenti 1 分 = 1 美分（固定锚定）→ 用户预期清晰
- 航空里程：浮动价值 → 用户算不清
- Mycelium 应该锚定 1 积分 = X？
- 如果让商家自定，会不会出现"通货膨胀"问题？

### Q.X4: 早期 lighthouse customer 选谁？

- 香港 / 大湾区 / 东南亚的哪类商家？
- 单店 vs 连锁？
- 高客单价 vs 低客单价？
- 数据成熟 vs 数据稀疏？

### Q.X5: SaaS / Performance / Settlement fee 三阶段过渡

- Phase 1（M1-M9）：Performance fee（不成交不收费）—— 验证假设
- Phase 1.5（M9-M18）：联盟匹配 + 服务费 —— 早期收入
- Phase 2（M18+）：SVF 持牌 + Settlement fee —— 长期商业模式

每个阶段的**触发条件**和**升级路径**？

### Q.X6: AI 优化器进入产品的时机

- M1：不上 AI（避免过度复杂）
- M3：商家级建议（避开 GDPR Art. 22）
- M9：跨商家联盟优化（数据聚合后才有价值）
- M18：完整 ROI 预测 dashboard

每个阶段的**最小可用功能**是什么？
